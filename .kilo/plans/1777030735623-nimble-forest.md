## PART 2: Docker Containerization Workflow

### Phase 5: Dockerfile Review and Optimization

The project already has a mature multi-stage Dockerfile. This phase reviews and optimizes it.

#### 5.1 Current Dockerfile Analysis

| Stage | Purpose | Base Image | Notes |
|-------|---------|------------|-------|
| base | Common base | node:20-alpine (pinned SHA) | Good: pinned digest |
| deps | Install deps | base | Good: npm ci --prefer-offline, BuildKit cache |
| builder | Build app | base | Uses STANDALONE_BUILD implicitly via npm run build |
| migrator | DB migrations | base | Has curl, postgresql-client, tsx |
| runner | Production runtime | base | Standalone output, health check, non-root user |

**Already implemented best practices:**
- Multi-stage build (reduces final image size)
- Pinned base image with SHA256 digest
- Non-root user (nextjs:1001)
- BuildKit cache mounts for npm and .next/cache
- npm ci --prefer-offline --ignore-scripts (deterministic installs)
- Package lock file verification
- NEXT_TELEMETRY_DISABLED=1
- HEALTHCHECK instruction
- Minimal runtime dependencies (curl, netcat, openssl)
- OCI image labels

**Issues to Address:**

1. **Duplicate header comments** (lines 1-45 are repeated)
2. **Standalone output not guaranteed**: next.config.ts only enables standalone when STANDALONE_BUILD=true OR non-Windows production. The Dockerfile does not set STANDALONE_BUILD=true as a build arg.
3. **Prisma generate runs twice**: Once in builder stage (npx prisma generate), and prisma generate is also in the build npm script. Redundant but harmless.
4. **netcat-openbsd in runner**: Only used for connectivity checks in entrypoint. Could use a lighter alternative or remove.

#### 5.2 Dockerfile Optimization Steps

1. **Remove duplicate header comments** (lines 1-45 duplicate lines 46-onward)

2. **Add STANDALONE_BUILD=true as a build arg** to ensure standalone output:
   - Add ARG STANDALONE_BUILD=true in builder stage
   - Add ENV STANDALONE_BUILD=true in builder stage

3. **Add a test stage** for running tests inside Docker:
   - FROM builder AS test
   - ENV NODE_ENV=test
   - RUN npx vitest run --coverage

4. **Pin Alpine version** for reproducibility:
   - Consider pinning to node:20.x.x-alpine3.19 for exact OS version

5. **Reduce migrator image size**: Remove curl if not needed (health checks are in runner only)

6. **Add labels for OCI compliance**:
   - LABEL org.opencontainers.image.source
   - LABEL org.opencontainers.image.revision (using GIT_SHA arg)

### Phase 8: Deployment Considerations

#### 8.1 Environment Configuration

| Environment | Compose File | Key Differences |
|-------------|-------------|-----------------|
| Local/Dev | docker-compose.yml | Exposed DB/Redis ports, relaxed security, SECURE_COOKIES=false |
| Production | docker-compose.prod.yml | No exposed DB/Redis ports, resource limits (512M app, 512M DB, 256M Redis), SECURE_COOKIES=true, LOCKOUT_FAIL_CLOSED=true, RATE_LIMIT_FAIL_CLOSED=true, internal-only network, read-only filesystem + tmpfs, required env vars with :? syntax |

**Required Production Environment Variables:**

- POSTGRES_PASSWORD - Strong database password
- JWT_SECRET - 64+ byte random key (openssl rand -base64 64)
- REDIS_PASSWORD - Strong Redis password
- SEED_ADMIN_PASSWORD - Override default admin password
- NEXT_PUBLIC_APP_URL - Public URL (e.g., https://stock.hay2010.com)

**Before deploying to production:**

1. Rotate all secrets from .env.docker (it was previously committed to git)
2. Use Docker Secrets or a secrets manager instead of env files
3. Set SECURE_COOKIES=true
4. Set LOCKOUT_FAIL_CLOSED=true (deny login if Redis unavailable)
5. Set RATE_LIMIT_FAIL_CLOSED=true (deny requests if Redis unavailable)

#### 8.2 Deployment Steps

**1. Pre-deployment validation:**

- npm run test:all
- npm run lint
- npx tsc --noEmit
- npm run test:security
- npm run security:validate-headers http://localhost:3000
- docker build --target test -t hay2010_stock:test .

**2. Build production image:**

- docker build --target runner --build-arg GIT_SHA=... --build-arg BUILD_DATE=... -t hay2010_stock:latest .
- Or: docker compose -f docker-compose.prod.yml build

**3. Push to registry (GHCR):**

- docker tag hay2010_stock:latest ghcr.io/org/hay2010_stock:latest
- docker push ghcr.io/org/hay2010_stock:latest

**4. Deploy:**

- docker compose -f docker-compose.prod.yml up -d
- docker compose -f docker-compose.prod.yml run --rm migrate
- SEED_ADMIN_PASSWORD=... docker compose -f docker-compose.prod.yml run --rm seed
- curl -f http://localhost:3000/api/health/public

**5. Post-deployment verification:**

- curl -s http://localhost:3000/api/health/public (check status)
- curl -sI http://localhost:3000/ | grep -E "Strict-Transport-Security|Content-Security-Policy" (security headers)
- docker compose -f docker-compose.prod.yml ps (service status)
- docker compose -f docker-compose.prod.yml logs --tail=50 app (recent logs)

#### 8.3 Rolling Updates / Zero-Downtime Deployment

For zero-downtime, use a reverse proxy (nginx/Caddy/Traefik) in front of the app container:

1. Start new container on different port
2. Update reverse proxy upstream
3. Drain old container (SIGTERM, graceful shutdown with 30s timeout)
4. Remove old container

The docker-entrypoint.sh already implements graceful shutdown with a 30-second timeout, which supports this pattern.

#### 8.6 Scaling Considerations

- **Horizontal scaling**: The app is stateless (JWT + Redis sessions), so multiple app containers can run behind a load balancer
- **Database**: Use read replicas for dashboard/report queries; connection pooler (PgBouncer) for high concurrency
- **Redis**: Use Redis Cluster or Sentinel for HA; the app already has lib/db/redis-cluster.ts
- **Background workers**: Run lib/workers as a separate container process for BullMQ workers
